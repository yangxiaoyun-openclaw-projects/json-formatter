# 规划与交办说明（PM → 开发 → 测试 → 发布）

本目录为**事实来源**（任务、状态、日志）；**交办在项目群里用 @ 完成**，被 @ 的 agent 会立刻被触发。

## 协作方式

**主方式：共享目录文件读写**。各角色更新 tasks、dev-log、qa-report、bugs，下一环节在心跳时自动检查并执行。**群 @ 为辅**：用户 @ 对应机器人可加速触发；机器人 @ 机器人无效。

## 任务状态流转（必遵守）

- **待开发** / **开发中** → 开发执行 → 完成后标记 **开发完成待测**（禁止直接标「已完成」）→ 测试心跳时自动检查
- **开发完成待测**：测试执行 → 通过则标 **测试通过**，不通过则写 bugs
- **待测试**：测试任务（负责人=测试团队），测试主动执行
- **测试通过** → 运维心跳时自动检查，用户确认后发布

## 开发交办规则

开发完成开发任务后**必须**：① 将任务状态改为「开发完成待测」；② 在 dev-log 记录。无需 @测试001，测试会在心跳时自动检查。

## 流程

1. **产品经理出规划**：写 `tasks/`，开发心跳时自动检查；用户可 @开发001 加速。
2. **开发**：实现并更新 dev-log、将任务改为「开发完成待测」；测试心跳时自动检查；用户可 @测试001 加速。
3. **测试**：测完更新 qa-report、bugs、将任务改为「测试通过」；运维心跳时自动检查。
4. **发布**：运维读 qa-report、用户确认后执行。

## 工作清单模式（无时间排期）

- 规划只写工作清单，不写时间排期。Agent 忘记时间，只按清单状态执行；每次被触发或心跳时检查共享工作区，有待办就做，持续高频检查。

## 工作痕迹

- 各 agent 在共享项目目录下维护 `AGENT_ACTIVITY.md`，只更新自己角色一节，在开始/进行中/阻塞/已完成时追加带时间的记录；用户可通过该文件确认谁在干活、卡在哪。
